DOCUMENTACION - FASE 2: FastAPI + Nginx (SOLO API, Cliente Qt)
Fecha: 2026-05-18 (actualizado)
Proyecto: 4eng-Login (Qt como único cliente, FastAPI backend)

==================================================
1) ¿QUE ES FASE 2?
==================================================
Tu proyecto tiene 2 fases claramente separadas:

FASE 1 (COMPLETADA): Qt Desktop + MySQL
- Objetivo: Autenticacion basica (login/registro)
- Responsables: Usuario con Qt en su PC
- Usuarios creados: Solo en tabla "users"
- Perfiles extendidos: VACIOS (no se tocan)
- Tablas activas: SOLO "users"
- Fin de fase: Usuario puede loguearse

FASE 2 (PROXIMA): FastAPI + Nginx (SOLO API)
- Objetivo: Backend centralizado, lógica de negocio y seguridad expuesta como API REST
- Responsables: Servidor en VPS
- Usuarios existentes: Los de tabla "users" (heredados de Qt)
- Perfiles extendidos: Se LLENAN aquí (engineer_profiles, company_profiles)
- Tablas activas: TODAS las 9 tablas
- Fin de fase: Sistema completo funcionando, Qt consume API (NO habrá frontend web)

==================================================
2) ¿POR QUE ESTAN SEPARADAS?
==================================================

Razon tecnica:
- Qt es DESKTOP (corre en tu PC Windows)
- FastAPI es BACKEND API (corre en VPS Linux)
- Qt se conecta a FastAPI usando HTTP (requests REST) a través de Nginx (proxy)
- NO habrá frontend web: Qt es el único cliente


Razon de negocio:
- Usuario final usa Qt (instalado en su PC) para login, registro, postulación, etc.
- Todas las operaciones avanzadas (perfiles, CVs, vacantes, postulaciones, IA, auditoría) se hacen desde Qt, que consume la API de FastAPI
- Admin y usuarios NO usan navegador, solo la app Qt


Razon de base de datos:
- La tabla "users" y las demás (engineer_profiles, company_profiles, etc.) son gestionadas por FastAPI
- Qt ya NO accede directo a MySQL/MariaDB: toda la lógica pasa por FastAPI

==================================================

3) QUE HACE FASTAPI EN FASE 2
==================================================
FastAPI es el backend central (API REST) que:

A) Recibe y autentica usuarios desde Qt (tabla users)
   - Qt envía usuario/password a FastAPI vía HTTP (no accede más directo a MySQL)
   - FastAPI valida credenciales, responde con JWT (token de sesión)
   - Hash de password: migración de SHA2 a Argon2id (o bcrypt)

B) Completa y gestiona perfiles (engineer_profiles, company_profiles)
   - Qt permite al usuario completar/editar su perfil y envía los datos a FastAPI
   - FastAPI guarda y valida la información

C) Gestiona CVs (cv_documents)
   - Qt permite subir PDF (usando multipart/form-data)
   - FastAPI recibe, parsea, guarda y asocia el CV

D) Gestiona vacantes (job_posts)
   - Qt permite a empresas crear/editar vacantes y a ingenieros verlas/postularse
   - FastAPI almacena y expone las vacantes

E) Procesa postulaciones (applications)
   - Qt permite postularse a vacantes
   - FastAPI registra la postulación

F) Evalúa con IA (ai_evaluations)
   - FastAPI puede llamar a OpenAI API para evaluar el match
   - Qt muestra el resultado al usuario

G) Sugerencias de capacitación (training_recommendations)
   - FastAPI sugiere cursos según brechas detectadas
   - Qt muestra sugerencias

H) Auditoría (admin_audit_logs)
   - FastAPI registra todas las acciones relevantes
   - Qt puede mostrar logs a usuarios con rol admin

4) FLUJO USUARIO (SOLO Qt)
==================================================
Escenario: Juan (ingeniero) postula a vacante de Empresa XYZ usando solo la app Qt

PASO 1: Juan se registra en Qt
- Abre app Qt en su PC
- Rellena: usuario=juan, pass=mipass, edad=28, lenguaje=Python
- Qt envía los datos a FastAPI vía endpoint /auth/register
- FastAPI crea usuario en tabla "users" (password hasheada)

PASO 2: Juan inicia sesión en Qt
- Ingresa usuario y contraseña en la app Qt
- Qt envía login a /auth/login
- FastAPI valida y responde con JWT

PASO 3: Juan completa perfil
- Qt permite completar perfil (años experiencia, skills, etc.)
- Qt envía datos a /profile/engineer
- FastAPI guarda en engineer_profiles

PASO 4: Juan sube CV
- Qt permite subir PDF
- Qt envía archivo a /cv/upload
- FastAPI procesa y guarda

PASO 5: Juan ve vacantes y postula
- Qt consume /jobs/list y /jobs/apply
- FastAPI registra postulación

PASO 6: IA evalúa candidato
- FastAPI llama a OpenAI y guarda resultado
- Qt muestra resultado al usuario

PASO 7: Admin audita
- Qt (solo admin) puede ver logs de acciones

==================================================

5) ¿POR QUÉ Qt SOLO USA LA API?
==================================================
Técnico:
- Qt solo interactúa con FastAPI vía HTTP (API REST), nunca accede directo a la base de datos
- FastAPI centraliza la lógica, seguridad y validaciones

Lógico:
- Todas las operaciones complejas (CVs, vacantes, IA, auditoría) se gestionan en el backend
- Qt solo muestra la UI y consume la API

Práctico:
- Permite evolucionar el backend sin cambiar Qt
- Seguridad: credenciales y lógica nunca expuestas al cliente

8) FLUJO DE RED EN FASE 2 (CLAVE)
==================================================
Toda petición de Qt sigue este camino:
1. Qt envía HTTP/HTTPS a Nginx (endpoint público)
2. Nginx recibe la ruta solicitada
3. Nginx enruta al servicio/puerto interno correspondiente (principalmente FastAPI)
4. FastAPI procesa la lógica y accede a MariaDB
5. La respuesta vuelve por Nginx a Qt

Regla de seguridad:
- Qt NO habla directo con FastAPI por puerto interno.
- Qt NO habla directo con MariaDB (3306).
- Nginx es la única puerta de entrada pública para la app Qt.

==================================================
9) HASH DE CONTRASEÑAS EN TRANSICIÓN
==================================================
FastAPI debe soportar transición gradual:
1. Intentar validar con password_hash (bcrypt/Argon2id)
2. Si password_hash es NULL, usar verificación legacy de users.password
3. Si login legacy es correcto, migrar inmediatamente a password_hash
