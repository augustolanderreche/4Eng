DOCUMENTACION TECNICA - BASE DE DATOS login_db
Fecha última actualización: 2026-05-15
Proyecto: 4eng-Login (preparado para evolucion a backend FastAPI + Nginx)

==================================================
1) OBJETIVO DE ESTA BASE
==================================================
Esta base modela una plataforma con 3 tipos de cuenta:
- Usuario (ingeniero)
- Empresa
- Admin

Cubre:
- autenticacion
- perfiles por rol
- CVs
- vacantes
- postulaciones
- evaluacion IA
- recomendaciones de capacitacion
- auditoria admin

==================================================
2) ARQUITECTURA GENERAL Y POR QUE ESTA ASI
==================================================
Diseno elegido: tabla central de usuarios + tablas especializadas por dominio.

Razon:
- Evita mezclar todos los datos en una sola tabla gigante.
- Permite crecer sin romper login/registro actual.
- Mantiene integridad con claves foraneas.
- Facilita que backend futuro aplique reglas por rol.

Flujo de negocio (resumen):
1. Se registra usuario o empresa en users.
2. Se completa perfil en engineer_profiles o company_profiles.
3. Empresa crea vacantes en job_posts.
4. Usuario sube CV (cv_documents) y postula (applications).
5. IA evalua postulacion (ai_evaluations) y sugiere capacitacion (training_recommendations).
6. Admin audita acciones sensibles (admin_audit_logs).

==================================================
3) TABLAS Y FUNCION DE CADA UNA
==================================================
A) users
- Proposito: autenticacion y datos base de cualquier cuenta.
- Campos clave:
  - id (PK)
  - role: Usuario | Empresa | Admin
  - username (UNIQUE)
  - email (UNIQUE)
  - password (texto plano, Qt lo usa actualmente)
  - password_hash (NULL por ahora, FastAPI lo usará con bcrypt)
  - is_active
  - created_at, updated_at
  - last_login (DATETIME NULL) ← NUEVO 2026-05-15, FastAPI lo actualizará al hacer login
- Índices activos (agregados 2026-05-15):
  - idx_users_username, idx_users_email, idx_users_role
- Por que existe:
  - Punto unico de identidad de cuenta para el sistema.

B) engineer_profiles
- Proposito: datos extendidos del rol Usuario (ingeniero).
- Relacion: user_id -> users.id (1 a 1)
- Por que existe:
  - Separar datos de perfil tecnico sin recargar users.

C) company_profiles
- Proposito: datos extendidos del rol Empresa.
- Relacion: user_id -> users.id (1 a 1)
- Por que existe:
  - Guardar info de negocio y contacto empresarial.

D) cv_documents
- Proposito: CVs subidos por ingenieros.
- Relacion: engineer_user_id -> users.id (1 a N)
- Campos utiles: file_url, file_sha256, extracted_text, parsed_skills(JSON)
- Por que existe:
  - Base de entrada para analisis IA y matching.

E) job_posts
- Proposito: vacantes creadas por empresas.
- Relacion: company_user_id -> users.id (1 a N)
- Campos utiles: required_skills(JSON), status, location_mode
- Índices activos (agregados 2026-05-15):
  - idx_job_posts_company, idx_job_posts_status
- Por que existe:
  - Modelo principal de demanda laboral.

F) applications
- Proposito: postulacion de un ingeniero a una vacante.
- Relaciones:
  - job_post_id -> job_posts.id
  - engineer_user_id -> users.id
  - cv_document_id -> cv_documents.id (nullable)
- Restriccion importante:
  - UNIQUE(job_post_id, engineer_user_id) evita postulacion duplicada.
- Índices activos (agregados 2026-05-15):
  - idx_applications_engineer, idx_applications_job, idx_applications_company, idx_applications_status
- Por que existe:
  - Vincular candidato + vacante + estado de proceso.

G) ai_evaluations
- Proposito: resultado del analisis IA por postulacion.
- Relacion: application_id -> applications.id
- Campos utiles: score, recommendation, missing_skills(JSON), raw_response
- Por que existe:
  - Trazabilidad de decisiones IA y explicabilidad.

H) training_recommendations
- Proposito: brechas y capacitaciones sugeridas por IA.
- Relacion: ai_evaluation_id -> ai_evaluations.id
- Por que existe:
  - Convertir brecha de skills en accion concreta para usuario.

I) admin_audit_logs
- Proposito: auditoria de acciones administrativas.
- Relacion: admin_user_id -> users.id
- Por que existe:
  - Seguridad, trazabilidad y cumplimiento.

==================================================
4) RELACIONES (MAPA RAPIDO)
==================================================
users (1) ---- (1) engineer_profiles
users (1) ---- (1) company_profiles
users (1) ---- (N) cv_documents
users (1) ---- (N) job_posts
users (1) ---- (N) applications [como engineer_user_id]
job_posts (1) - (N) applications
cv_documents (1) - (N) applications (opcional)
applications (1) - (N) ai_evaluations
ai_evaluations (1) - (N) training_recommendations
users (1) ---- (N) admin_audit_logs [como admin_user_id]

==================================================
8) COMO SENSIBILIZAR/PROTEGER DATOS SENSIBLES
==================================================
8.1 Contrasenas (obligatorio)
Plan recomendado:
1. Dejar de usar users.password para login real.
2. Usar users.password_hash para almacenar hash.
3. El backend FastAPI debe:
   - al registrar: hash(password) y guardar en password_hash
   - al login: verify(password_ingresada, password_hash_guardado)

Algoritmos recomendados:
- Argon2id (preferido)
- bcrypt (alternativa valida)

8.2 Migracion de datos de prueba (paso inicial)
- Cambiar las 3 cuentas de prueba por hashes.
- Luego vaciar o eliminar columna password cuando el frontend/backend ya no la use.

8.3 Minimo de privacidad en consultas
- No hacer SELECT * en endpoints publicos.
- Devolver solo campos necesarios.

8.4 Enmascarar datos en paneles
Ejemplo de email enmascarado:
- usuario@mail.com -> u***o@mail.com

Ejemplo SQL de vista enmascarada basica:
CREATE OR REPLACE VIEW v_users_masked AS
SELECT
  id,
  role,
  username,
  CONCAT(LEFT(email,1), '***', SUBSTRING(email, LOCATE('@', email)-1)) AS email_masked,
  display_name,
  is_active,
  created_at
FROM users;

Nota:
El enmascarado en SQL ayuda para vistas admin; la proteccion real de contrasena es hash.

8.5 Cifrado en transito
- Activar HTTPS para phpMyAdmin (dominio + certbot) para no enviar credenciales en texto claro.

8.6 Principio de minimo privilegio
- backend_app: sin permisos DDL (no CREATE/DROP).
- dbadmin: solo para administracion.
- No usar root para app ni para uso diario.
